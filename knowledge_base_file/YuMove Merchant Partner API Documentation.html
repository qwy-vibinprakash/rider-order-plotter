YuMove Merchant Partner API Documentation

version 1.1
The YuMove Merchant Partner API enables Merchants and Enterprises to integrate with the YuMove Network to power their last-mile deliveries. Merchants and Enterprises use the API to automate fulfillment workflows and manage delivery operations efficiently, including checking serviceability, creating and managing orders, receiving real-time status updates, and tracking deliveries throughout their lifecycle.
NOTE: Access to YuMove Merchant Partner API is only available for YuMove Merchant Partner clients. Please reach out to our team for more information and to get access.

Authentication

To authenticate and access YuMove APIs, the following headers are required for each request:
X-REQUEST-ID: A unique ID generated by the party for the first serviceability API call. This ID should be used for every subsequent request until the end of the order process.
X-API-KEY: An API key generated and provided by YuMove for each individual party
Additionally, If the partner opts for SHA256 Checksum Method, they need to pass;
X-API-TOKEN : A token generated using the X-REQUEST-ID and a secret token provided by YuMove during the onboarding phase.
X-API-TOKEN = sha256(X-REQUEST-ID + secret_token)
​

Order Category

YuMove supports multiple order categories. Each category has its own operational rules, fulfilment type compatibility, and serviceability constraints.
Food – Prepared meals from restaurants, cafés, or cloud kitchens.
Fashion – Apparel, footwear, and accessories.
Grocery – Groceries and daily essentials.
Pharma – Medicines and healthcare products.
More categories will be added over time.

Service Types

Store Pick-up (Standard)
Orders are assigned to a rider upfront, who goes to the store to collect a specific prepared order. The rider is tied to that order from the beginning, ensuring a direct and predictable pickup flow.
FIFO (YuFirst)
Virtual orders are created and broadcast by the merchant. When riders check in, they join a queue, and the first rider is assigned the first ready order. The actual order and drop details are then updated, minimising rider wait times and enabling faster dispatch, ideal for quick commerce.

Service Type

Current service type supported.
Service Type
Description
STORE_PICKUP
Orders are assigned to a rider upfront, who goes to the store to collect a specific prepared order. The rider is tied to that order from the beginning, ensuring a direct and predictable pickup flow.
FIFO
Orders are prepared and broadcasted by the merchant. Rider mapping happens when riders check in, with the first rider getting the first ready order. This minimizes waiting time, ideal for quick commerce.
